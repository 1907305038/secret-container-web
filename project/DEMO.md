## 1. 系统总览 — TDX/SGX 硬件 + K8s 集群状态

### 页面

访问 `http://IP:8080/`（主页）

### 功能说明

展示集群级别的汇总信息，包括 Intel TDX/SGX 硬件状态、K8s 集群版本、Pod 运行情况、运行时分类和命名空间分布。

## 2. 机密容器创建与销毁的直观展示

### 页面

访问 `http://IP:8080/pods`

### 功能说明

- 创建 Pod 后，通过 **WebSocket 实时推送** Pod 状态变化（Pending → ContainerCreating → Running）
- 卡片上出现**阶段徽章**，实时反映当前状态
- 删除 Pod 时卡片执行**缩放淡出动画**（0.4 秒）

### 演示步骤

#### 2.1 快速创建 TDX Pod

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 点击「🛡️ 快速 TDX」按钮 | 自动填入表单：运行时 `kata-qemu-tdx`、镜像 `nginx:alpine`、名称 `tdx-<随机>` |
| 2 | 点击「🚀 创建 Pod」 | 顶部提示 `✅ tdx-xxx 已创建，等待启动...` |
| 3 | **观察卡片** | 新卡片出现，徽章依次变化 |
| 4 | 等待约 10-30 秒 | 卡片状态稳定为 🟢 Running |

#### 2.2 观察阶段徽章变化

| 阶段徽章 | 外观 | 含义 |
|----------|------|------|
| `PENDING` | 橙色背景 `#fff3e0` | Pod 已提交，等待调度 |
| `CONTAINERCREATING` | 蓝色脉冲 `#e3f2fd` | 正在拉取镜像 / 创建容器 |
| (无徽章) | 绿色实心圆点 | Pod 进入 Running，正常运行 |

```
┌──────────────────────────────────────────────────────────┐
│ 🟢 TDX  tdx-m8rk2  default  PENDING  ● Pending   📋  ▸ │  ← 阶段1
│ 🟢 TDX  tdx-m8rk2  default  CONTAINERCREATING  📋  ▸   │  ← 阶段2 (蓝色脉冲)
│ 🟢 TDX  tdx-m8rk2  default  ● Running            📋  ▸ │  ← 阶段3 (稳定)
└──────────────────────────────────────────────────────────┘
```

#### 2.3 自定义创建 Pod

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 点击「➕ 创建 Pod」 | 弹出模态表单 |
| 2 | 在「📦 基本配置」中输入：名称、命名空间、镜像、运行时 | 表单字段可编辑 |
| 3 | 切换到「⚙️ 资源 & 高级」标签 | 显示 CPU/内存请求和限制 |
| 4 | 点击预设按钮「💡 小型」/「📦 中型」/「🚀 大型」 | 自动填充资源参数 |
| 5 | 点击「🚀 创建 Pod」 | 提交创建，返回结果 |

#### 2.4 删除 Pod（观察动画）

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 鼠标悬停目标 Pod 卡片 | 卡片右侧出现 🗑️ 删除按钮 |
| 2 | 点击 🗑️ 按钮 | 顶部提示 `🗑️ xxx 已删除` |
| 3 | **观察卡片** | 卡片执行缩放(0.95→) + 透明度(1.0→0.3) 淡出，0.5 秒后从列表移除 |
| 4 | 列表自动刷新 | 总数更新 |

| 3 | 观察时间线内容 | 每个事件显示：时间戳、原因、详细消息 |

### 预期时间线示例

```
📜 生命周期

 │

 │
```

## 5. 内存加密验证（核心功能）

### 页面

访问 `http://IP:8080/pods`，展开任意 TDX Pod

### 功能说明

通过对比**宿主机读 QEMU 进程内存**和**容器内读自身内存**，直观展示 TDX 内存加密：

- **宿主机视角**：读到的 QEMU 内存是 TDX 加密后的密文（乱码，高熵值）
- **容器内视角**：读到的内存是明文（可读，低熵值）

### 5.1 全自动模式（推荐）

#### 原理

```
1. 后端在容器内执行: echo "TDX_VERIFY_<timestamp>" > /dev/shm/proof.txt
2. 容器内读取 → 明文: "TDX_VERIFY_1752123456"
3. 宿主机读取 QEMU 进程的 /proc/<qemu_pid>/mem
4. 在 QEMU 内存中搜索明文 → 找不到！（TDX 加密了）
5. 展示宿主机内存区域的 hex dump → 乱码密文
```

#### 演示步骤

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 展开任意 TDX Pod 详情 | 滚动到底部 |
| 2 | 找到「🔐 内存加密验证」面板 | 显示两个按钮 |
| 3 | 点击「🔄 一键验证 (全自动)」 | 按钮变为 `⏳ 验证中...` |
| 4 | 等待 3-5 秒 | 显示对比结果 |

#### 预期结果

```
🔐 内存加密验证
测试明文: TDX_VERIFY_1752123456          [🔄 一键验证] [🔍 手动对比]

┌── 🖥️ 宿主机视角 (Host) ──────┐  ┌── 📦 容器内视角 (Guest) ──────┐
│ QEMU PID: 123456               │  │ kubectl exec                     │
│                                │  │                                  │
│ ┌ heap 7f4a0000-7f4a1000 ───┐ │  │ ┌ guest /dev/shm ─────────────┐ │
│ │ 00000000 8B 3C A1 F2 9E 4D│ │  │ │ 00000000 54 44 58 5F 56 45│ │
│ │ 00000010 7C 03 66 B2 D8 11│ │  │ │ 00000010 52 49 46 59 5F 31│ │
│ │ 00000020 E5 AA 47 F3 19 2C│ │  │ │ 00000020 37 35 32 31 32 33│ │
│ │ 00000030 81 4F 92 6E 33 7A│ │  │ │ 00000030 34 35 36 00 00 00│ │
│ │        |.<.ÞM|.fÒ..ªG..,|│ │  │ │        |TDX_VERIFY_175212│ │
│ │                            │ │  │ │        |3456...           │ │
│ │ 🔴 熵值: 7.92 (密文特征)   │ │  │ │ 🟢 熵值: 4.51 (明文特征)  │ │
│ └────────────────────────────┘ │  │ └──────────────────────────────┘ │
│                                │  │                                  │
│ ✅ 宿主机内存中未找到明文       │  │ ✅ 容器内明文完整可见             │
│   (已搜索 2 个区域)             │  │                                  │
└────────────────────────────────┘  └──────────────────────────────────┘
```

#### 关键指标解读

| 指标 | 宿主机 (Host) | 容器内 (Guest) | 解读 |
|------|---------------|----------------|------|
| **熵值** | ~7.9 (高) | ~4.5 (中低) | 密文接近随机(熵≈8)，明文含结构化信息(熵低) |
| **明文搜索** | 未找到 | 完整可见 | TDX 加密后宿主机无法搜索到明文 |
| **Hex dump** | 随机字节 | 可读 ASCII | 左边乱码、右边可读 |
| **边框颜色** | 🔴 红色 | 🟢 绿色 | 密文区域红色警示，明文区域绿色安全 |

### 5.2 半自动模式

#### 原理

不自动写入测试数据，只读取宿主机侧 QEMU 进程的 `/proc/<pid>/mem` 内容，展示 hex dump 和熵值分析。

#### 演示步骤

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 在「🔐 内存加密验证」面板中 | 点击「🔍 手动对比 (半自动)」 |
| 2 | 等待 1-2 秒 | 显示宿主机 QEMU 内存区域 |
| 3 | 观察 hex dump | 看到乱码字节 + 高熵值 |

#### 半自动 vs 全自动对比

| 模式 | 写入测试数据 | 容器内读取 | 宿主机读取 | 适用场景 |
|------|-------------|-----------|-----------|----------|
| 全自动 | ✅ 自动写入 | ✅ 自动读取 | ✅ 自动读取 | 演示/快速验证 |
| 半自动 | ❌ 不写入 | ❌ 不读取 | ✅ 读取 QEMU 内存 | 手动测试/自定义数据 |

#### 半自动提示

半自动模式下，页面会显示操作提示：
```
💡 半自动模式：请在容器内执行 echo "TEST" > /dev/shm/proof.txt 后使用全自动验证
```

## 8. WebSocket 实时事件

### 连接端点

`ws://IP:8080/ws/state`

### 功能说明

前端在 Pod 页面建立 WebSocket 长连接，接收服务器推送的实时事件，无需轮询。

### 事件类型

| 事件类型 | JSON 示例 | 触发条件 |
|----------|-----------|----------|
| `pod_created` | `{"type":"pod_created","name":"tdx-xxx","namespace":"default","runtime":"kata-qemu-tdx","image":"nginx:alpine"}` | 创建 Pod 成功后 |
| `pod_deleted` | `{"type":"pod_deleted","name":"tdx-xxx","namespace":"default"}` | 删除 Pod 成功后 |
| `pod_phase` | `{"type":"pod_phase","name":"tdx-xxx","namespace":"default","phase":"Running"}` | Pod 状态变更（每 10s 检测） |
| `pod_count` | `{"type":"pod_count","count":11,"message":"pod 状态已更新"}` | Pod 总数变更 |
| `tdx_status` | `{"type":"tdx_status","data":{...}}` | TDX 启用状态变更 |

### 前端行为

| 收到事件 | UI 反应 |
|----------|---------|
| `pod_created` | 顶部 toast：`✅ tdx-xxx 已创建，等待启动...`，2 秒后刷新列表 |
| `pod_deleted` | 顶部 toast：`🗑️ xxx 已删除`，卡片执行淡出动画，0.5 秒后移除 |
| `pod_phase` | 卡片上出现阶段徽章（PENDING / CONTAINERCREATING / 消失=RUNNING） |
| `pod_count` | 自动刷新 Pod 列表和统计数字 |
| `tdx_status` | 更新硬件状态卡片 |

### WebSocket 演示

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 打开两个浏览器标签页到 `/pods` | 两个页面都建立 WebSocket |
| 2 | 在标签页 A 中创建一个 Pod | 标签页 B 也收到 `pod_created` 并自动刷新 |
| 3 | 在标签页 B 中删除一个 Pod | 标签页 A 也收到 `pod_deleted` 并展示动画 |
| 4 | 断开网络再恢复 | WebSocket `onclose` 后 5 秒自动重连 |
